=== SAME-DAY CORRECTION (2026-07-19, later this session) ===
The "~157 files dead" claim above was investigated further while planning the actual cleanup
and found FALSE. A rigorous reachability audit (import BFS + Hilt @IntoSet multibinding trace,
hand-verified against source) found only 29 files are genuinely dead. The other 128 are live:
- ui/modelmanager/ModelManagerViewModel.kt + ui/benchmark/* are directly wired into
  MainActivity.kt/RelaisAppShell.kt (not through GalleryNavGraph at all).
- customtasks/agentchat/ (skill manager + URL-based skill install, MCP client w/ OAuth+header
  auth, WebView agent sandbox), customtasks/mobileactions/, customtasks/tinygarden/,
  ui/llmchat/, ui/llmsingleturn/ are all Hilt @Provides @IntoSet multibindings feeding
  ModelManagerViewModel's `Set<CustomTask>` constructor parameter — a wiring mechanism that
  has ZERO Kotlin import connecting it to RelaisAppShell, so a plain grep/BFS (which is what
  produced the original "157 dead" claim) cannot see it. Deleting these would break the build
  via Hilt annotation processing.
- customtasks/agentchat/ in particular has NO replacement anywhere in the new Relais-native
  stack (nodetools/ is a fixed 4-tool list; no user-facing skill/MCP management exists) — so
  even setting DI-liveness aside, deleting it would be a real feature regression, not cleanup.
docs/CODEMAPS/architecture.md and frontend.md have been corrected same-day. The actual cleanup
plan (scoped to the verified 29-file dead list only) is at
.claude/PRPs/plans/dead-code-cleanup-ui-customtasks.plan.md.
LESSON: for "is this code reachable" questions in Hilt/Dagger codebases, multibinding
(@IntoSet/@IntoMap) is invisible to import-graph analysis — always grep for @IntoSet/@IntoMap
providers separately and trace their consumers before declaring code dead.
